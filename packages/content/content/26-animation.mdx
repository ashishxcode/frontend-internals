## The Problem That Hooks You

You add a hover effect. The box moves right when the user hovers. You use `left` and `width` because they seem natural. The animation is choppy. The box stutters. CPU spikes. Users see jank.

Or you build a modal that should fade out when closed. You toggle a state variable. The modal disappears instantly. No exit animation. The user is confused about where the modal went.

Or you build a list that reorders. Items jump to new positions instead of sliding smoothly.

These are the three animation problems developers face daily: janky motion, missing exit transitions, and layout jumping.

## Why It Happens

Naive animation uses the obvious CSS properties that describe position and size: `left`, `top`, `width`, `height`, `margin`. These trigger the browser's layout step on every animation frame. For the full rendering pipeline explanation, see Ch 07.

The key insight for animation: `transform` and `opacity` skip layout and paint entirely, running only on the compositor thread.

For exit animations, React unmounts components the moment the condition becomes false. There is no hook for "wait, animate first, then unmount." Without a tool that delays removal, exit animations are impossible.

## The One Insight

**Smooth animation is about picking the right properties to animate.** Only `transform` and `opacity` are compositor-friendly. Everything else triggers layout or paint. That's the whole secret.

Think of it like a painter. The compositor thread is like a separate artist who can move, rotate, and fade pre-painted layers on a canvas without asking the main artist to repaint anything. If you ask the main artist to repaint every frame (layout properties), you get lag. If you let the compositor artist do the moving (transform/opacity), it's smooth.

A library like Framer Motion is a declarative layer over this truth. You declare start and end states. It interpolates compositor properties. It handles enter, exit, and layout transitions for you.

Animation is communication. It shows continuity, cause, and effect. A button that moves down when pressed communicates that it was pressed. A modal that slides in from the right communicates its origin. Animation must be purposeful and must respect users who cannot tolerate motion.

## Three Scenarios

**Scenario 1: move a card on hover the expensive way.**

```css
.box { transition: left 300ms, width 300ms; }
.box:hover { left: 200px; width: 400px; }
```

Internally, the browser sees `left` change. It marks the layout as dirty. Each frame of the transition runs the full pipeline: layout recalculates positions, paint paints new pixels, composite layers them. With many elements, each layout recalculates the entire subtree. Frames drop.

**Scenario 2: move the same card with `transform`.**

```css
.box { transition: transform 300ms ease, opacity 300ms; }
.box:hover { transform: translateX(200px) scale(1.1); }
```

Internally, the browser sees `transform` change. It marks only the composite step as dirty. The compositor thread on the GPU applies the transformation as a matrix multiplication. No layout. No paint. This stays smooth even at 60fps with hundreds of elements.

**Scenario 3: exit animation without and with `AnimatePresence`.**

Without `AnimatePresence`, when `isOpen` becomes `false`, React removes the modal from the virtual DOM immediately. The next commit removes the DOM node. The modal vanishes. No transition runs.

With `AnimatePresence`, Framer Motion intercepts the removal. It keeps the element in the DOM. It changes the animation target to the `exit` props and starts the animation. When the animation completes, Framer Motion signals React to remove the element.

## How the Compositor Works

The browser compositor is a separate thread from the main JavaScript thread (see Ch 07 for full pipeline details). When you change `transform`, the main thread marks the layer as dirty. The compositor thread reads the new value, applies it using the GPU, and composites the frame. The main thread never recalculates layout or repaints.

When you change `left`, the main thread must recalculate layout. Layout is synchronous and blocking. The main thread walks the DOM tree, computes new geometries, and marks paint regions. Then paint runs. Only then does the compositor get new bitmaps to composite.

The FLIP technique that Framer Motion uses for `layout` animations works like this:
- **First**: record the element's current position and size.
- **Last**: record the element's new position after layout change.
- **Invert**: calculate the difference. Apply a `transform` to make the element appear in its old position.
- **Play**: animate the `transform` to zero, so the element slides to its new position.

All steps except the initial layout read use compositor-only properties. The `layout` prop does this automatically.

## Real World: Notification Toast System

You build a notification toast system. Toasts appear from the top right, stay for 3 seconds, then disappear.

```jsx
import { motion, AnimatePresence } from "framer-motion";

function ToastContainer({ toasts, onRemove }) {
  return (
    <div className="toast-container">
      <AnimatePresence>
        {toasts.map(toast => (
          <motion.div
            key={toast.id}
            initial={{ opacity: 0, x: 100 }}
            animate={{ opacity: 1, x: 0 }}
            exit={{ opacity: 0, x: 100, transition: { duration: 0.15 } }}
            transition={{ duration: 0.3 }}
            className="toast"
          >
            {toast.message}
            <button onClick={() => onRemove(toast.id)}>x</button>
          </motion.div>
        ))}
      </AnimatePresence>
    </div>
  );
}
```

What happens internally: When a toast is added, Framer Motion creates the element with `opacity: 0` and `transform: translateX(100px)`. On the next frame, it animates to `opacity: 1` and `transform: translateX(0)`. The browser composites these on the GPU. When the toast is removed, `AnimatePresence` keeps the element mounted. Framer Motion animates back to the exit values. After 150ms, it signals removal.

Add accessibility: Use `useReducedMotion()` or a CSS media query to disable motion when the user prefers reduced motion.

## Tradeoffs

**CSS transitions vs Framer Motion.** CSS transitions work for simple state changes. They're lightweight and run off the main thread. But they can't orchestrate complex sequences, spring physics, or exit animations. Use CSS for simple hovers and toggles. Use Framer Motion for rich interactions and exit animations.

**requestAnimationFrame vs setInterval.** rAF aligns with the browser's paint cycle. It fires before the next paint, so you write one frame of changes at the right time. setInterval fires on a timer that drifts and has no relation to the paint cycle. Never use setInterval for animation.

**Opacity vs visibility.** `opacity` is compositor-friendly. `visibility` triggers paint. If you need to hide something smoothly, animate `opacity` to zero.

**transform vs geometry.** `transform: translateX()` creates the same visual result as `left` for position changes. But `transform` is compositor-only (Ch 07 explains why). Always prefer `transform` for position.

## Common Mistakes

- Animate `top`, `left`, `width`, `height`, or `margin`. These trigger layout per frame.
- Use `setInterval` for animation. Off-frame timing and drift.
- Forget exit animations need `AnimatePresence`. React unmounts instantly.
- Ignore `prefers-reduced-motion`. Motion makes some users physically ill.
- Over-animate. Motion distracts instead of communicating.
- Use `display: none` to hide animated elements mid-transition.

## Follow-up Questions

**Q1: What is the rendering pipeline and which stages does `transform` skip?**
The browser rendering pipeline has four stages: **Style** (matching CSS selectors to elements), **Layout** (calculating geometry — position, size), **Paint** (filling pixels — colors, shadows, text), and **Composite** (layering painted bitmaps on the GPU). `transform` and `opacity` skip **Layout** and **Paint** entirely. When you change `transform`, the browser marks only the composite step as dirty. The compositor thread on the GPU applies the matrix multiplication without recalculating geometry or repainting pixels. This is why `transform: translateX(100px)` is always smoother than `left: 100px` — the latter triggers all four stages on every frame.

```
Style → Layout → Paint → Composite
  ↓       ↓       ↓        ↑
  skip    skip    skip   transform/opacity only
```

**Q2: Why does animating `width` on a flex child cause other elements to move?**
`width` triggers the **Layout** stage. When a flex child's width changes, the browser recalculates the geometry of the entire flex container — including the position of every sibling. Each frame of the animation runs this full recalculation, causing siblings to shift as the width interpolates. `transform: scaleX()` avoids this because it skips Layout. The element visually stretches without changing its layout box. Siblings stay in place because the browser's layout model doesn't know the transform happened — only the compositor sees it.

```css
/* This causes sibling movement — layout recalculates every frame */
.flex-child { transition: width 300ms; }
.flex-child:hover { width: 200px; }

/* This is smooth — no layout, no sibling movement */
.flex-child { transition: transform 300ms; }
.flex-child:hover { transform: scaleX(1.5); }
```

**Q3: Walk through the FLIP steps and explain why it stays on the compositor.**
FLIP stands for **First, Last, Invert, Play**: (1) **First**: Record the element's current position and size before the layout change (one layout read). (2) **Last**: Let the layout change happen normally — the browser computes the new position. (3) **Invert**: Calculate the delta between old and new positions. Apply a `transform` that moves the element *back* to its old position — so it visually stays put despite the DOM reorder. (4) **Play**: Animate the `transform` to `none` (or zero), smoothly sliding the element to its new position. Steps 3 and 4 use only `transform`, which is compositor-friendly. The only layout read happens once in step 1. All animation frames run on the compositor thread without triggering reflow or repaint.

```jsx
// Framer Motion's layout prop does FLIP automatically
<motion.div layout>{content}</motion.div>
```

**Q4: A user sets `prefers-reduced-motion: reduce`. How do you handle it without rewriting every component?**
Two approaches: (1) **Wrap the animation library's motion components.** Create a wrapper like `MotionDiv` that checks `useReducedMotion()` from Framer Motion. If the user prefers reduced motion, override `initial`, `animate`, and `exit` to `{ opacity: 1 }` (no motion, instant appearance). All components using `MotionDiv` automatically respect the preference.

```jsx
function MotionDiv({ initial, animate, exit, ...props }) {
  const prefersReduced = useReducedMotion();
  return (
    <motion.div
      initial={prefersReduced ? false : initial}
      animate={prefersReduced ? false : animate}
      exit={prefersReduced ? false : exit}
      {...props}
    />
  );
}
```

(2) **CSS approach**: Add a global rule `*, *::before, *::after { animation-duration: 0.01ms !important; transition-duration: 0.01ms !important; }` inside a `@media (prefers-reduced-motion: reduce)` block. This disables all CSS transitions and animations globally without touching component code.

**Q5: You have a drag-and-drop grid where items swap positions. After the animation, DOM order must match visual order for keyboard users. How?**
Use Framer Motion's `layoutId` to animate position changes while keeping the DOM order intact until the animation completes. Each item gets a stable `layoutId` based on its data ID. When items swap, Framer Motion interpolates their positions using FLIP. After the animation finishes, commit the actual DOM reorder in a `useLayoutEffect` or `onAnimationComplete` callback.

```jsx
{items.map(item => (
  <motion.div
    key={item.id}
    layoutId={item.id}
    onAnimationComplete={() => {
      // Commit DOM reorder after animation finishes
      reorderDOM(items);
    }}
  >
    {item.content}
  </motion.div>
))}
```

Alternatively, maintain the visual order in state and the DOM order separately — animate the visual positions, then sync the DOM after the transition. The key constraint: keyboard tab order follows DOM order, not visual order, so the DOM must eventually match the visual layout.
