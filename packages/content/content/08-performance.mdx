## The one mental model

> **Performance is one question asked at three layers: "how do I do LESS work?" You either
> (1) do less work now — render fewer things (virtualization), skip wasted renders (memo),
> ship less JS (code-splitting); (2) do it later — defer/chunk/lazy so the main thread stays
> free for paint and input; or (3) do it elsewhere — off the main thread (Web Workers) or
> ahead of time (build/server). And you NEVER guess — you measure first, because the
> bottleneck is rarely where you think.**

Everything below is an instance of less / later / elsewhere. You won't memorize a checklist of
tricks — you'll classify any perf problem into one of those three and pick the matching tool.

## Key Mental Models

- **The main thread is the scarce resource** (Ch 02). Anything hogging it costs you paint/input.
- **Render only what's visible.** A list of 500k rows should mount ~20 DOM nodes, not 500k.
- **Memo trades compare-cost for skipped-render-cost.** Only a win when the skip is bigger.
- **Measure → fix → re-measure.** Profiler/Performance panel, not vibes.

## Introduction

At SDE-2 the perf conversation is never "name optimizations." It's "here's a heavy page, reason
about where the time goes and what you'd change." Interviewer's version is a contacts table up to
**5 lakh rows** with real-time updates. The whole thing is solvable from "do less work, measure
first." We build that reflex.

---

## Problem

Naively rendering a big list:

```jsx
{contacts.map(c => <Row key={c.id} contact={c} />)}   // 500,000 <Row>s
```

This creates 500k Fibers and 500k× DOM nodes. The render phase (Ch 04) is a giant task → the
tab freezes (Ch 02); the commit inserts hundreds of thousands of nodes → layout/paint chokes
(Ch 07); memory balloons. But the screen only shows ~20 rows. **499,980 rows are pure waste.**
The problem is doing work proportional to the *data* instead of proportional to the *viewport*.

---

## Mental Model — virtualization (windowing)

Render only the rows in (or near) the viewport; as the user scrolls, recycle the same handful
of DOM nodes to show different data. A tall empty spacer fakes the full scroll height.

```
   full list = 500,000 rows           rendered DOM = ~visible + overscan
   ┌───────────────┐  scrollTop        ┌───────────────┐
   │   (spacer top)│ ───────────────▶  │ Row 7000      │ ◀ startIndex = scrollTop / rowHeight
   │               │                   │ Row 7001      │
   │  [ viewport ] │  only these       │ ...           │
   │               │  exist in DOM     │ Row 7020      │ ◀ endIndex
   │ (spacer below)│                   └───────────────┘
   └───────────────┘                   positioned with transform: translateY(7000*rowHeight)
```

```mermaid
flowchart LR
  scroll[onScroll: read scrollTop] --> calc["start = floor(scrollTop/rowH)<br/>end = start + visibleCount + overscan"]
  calc --> slice["render contacts.slice(start, end)"]
  slice --> pos["offset each row via transform: translateY"]
  pos --> spacer["container height = total*rowH (fake full scroll)"]
```

Core math: `startIndex = Math.floor(scrollTop / rowHeight)`, render `visibleCount + overscan`
rows, position the window with `transform: translateY(startIndex * rowHeight)`. Use
`@tanstack/react-virtual` in real life; know the math for interviews.

### Why `transform`, not `top`/absolute (Ch 07 payoff)
Positioning rows with `top`/`left` triggers **layout + paint** every scroll frame (expensive).
`transform: translateY()` is **composited on the GPU** — it skips layout and paint entirely. On
a list updating every scroll frame this is the difference between smooth and janky.

### Virtualization tradeoffs (recite these — they ask explicitly)
- **Ctrl-F / native find breaks** — off-screen rows aren't in the DOM.
- **Accessibility** — screen readers need `aria-rowcount`/`aria-setsize`; focus on an
  unmounted row is lost.
- **Variable/unknown row heights** — need measurement or estimates; wrong heights = scroll jump.
- **Sticky headers, "scroll to row N", scroll restoration** — all need explicit handling.
- **SEO** — virtualized content isn't crawlable (irrelevant for an authed app like this).

## Engine Simulation — wasted renders & memo

```jsx
function Table({ rows, onRowClick }) {
  return rows.map(r => <Row key={r.id} row={r} onClick={onRowClick} />);
}
```

One row's status updates → `rows` array changes → `Table` re-renders → **every** `<Row>` re-runs
(Ch 03 top-down). For 20 visible rows that's fine; the cost is when a row is expensive or updates
fire constantly. Two derived fixes:

1. **`React.memo(Row)`** — skip a row whose props didn't change (`Object.is` shallow compare).
   But it only works if props are referentially stable — so `onClick` must be stable:
2. **`useCallback(onRowClick, [])`** — stabilize the function identity (Ch 01: a new function
   each render = new address = memo sees "changed"). Now unchanged rows skip re-render.

```
without memo:  1 status event → 20 Row renders
with memo + stable props:  1 status event → 1 Row render (the changed one)
```

**The trap (straight from Interviewer — see interview guide):** do NOT `memo` everything. `memo` stores prev
props and runs a compare every render; for cheap components that's slower than just rendering.
And it silently no-ops when props are inline objects/functions. The *structural* fix often beats
memo: push state down, or pass expensive subtrees as `children` so their element identity is
stable across the parent's state changes. **Memo measured hot spots; prefer composition.**

## Do-later & do-elsewhere

- **Code-splitting (ship less now):** `const Settings = lazy(() => import('./Settings'))` +
  `<Suspense>`. Route-level splitting means the contacts page doesn't ship the settings bundle.
  Vite/Rollup tree-shake unused exports (Ch 20).
- **Defer non-urgent renders (do later):** `startTransition`/`useDeferredValue` (Ch 04) keep
  typing responsive while an expensive filtered list renders at low priority.
- **Off the main thread (do elsewhere):** heavy parsing/sorting of a big dataset → a **Web
  Worker** (Ch 17), so the 200ms compute doesn't freeze scroll/paint (Ch 02).
- **Debounce/throttle input:** a search box hitting the server on every keystroke → debounce
  (do less, later). Throttle scroll handlers.

---

## Measure first — the tools

- **React DevTools Profiler:** flame graph of what rendered and why ("Why did this render?").
  Find wasted renders before adding memo.
- **Chrome Performance panel:** long tasks (Ch 02), layout/paint (Ch 07), main-thread time.
- **Core Web Vitals (real-user grades):**
  - **LCP** (Largest Contentful Paint) — load: largest element painted. Image/bundle work.
  - **INP** (Interaction to Next Paint) — responsiveness: how fast UI reacts to input.
    Long tasks and heavy renders hurt it — the whole reason for transitions/workers.
  - **CLS** (Cumulative Layout Shift) — stability: content jumping (reserve space for async).

## Interview Discussion (reason first)

**Q1. "500k-row table — how do you make it not die?"**
> "Render proportional to the viewport, not the data: **virtualize** — mount ~20 rows + overscan,
> recycle on scroll, position with `transform`, spacer for scroll height. Fetch pages on demand
> (cursor-based), don't hold 500k in memory; cache with TanStack (Ch 10). Server-side
> search/filter/sort. Then measure with the Profiler and only memoize rows that prove costly."

**Q2. "Should you memo every component?"**
> "No. Memo costs a prop compare and storage; for cheap components that's a net loss, and it
> no-ops when props aren't stable. I memo measured hot spots and prefer structural fixes
> (state colocation, `children` as props). Most re-renders are harmless." *(Don't fall into the
> 'so your app does tons of unnecessary renders' trap — say they're cheap and you fix measured ones.)*

**Q3. "Downside of virtualization?"**
> Ctrl-F breaks, a11y needs extra ARIA, variable heights cause jumps, sticky/scroll-to-row need
> handling. The `top`-vs-`transform` reflow point (Ch 07).

*Scoring:* full = viewport-not-data + transform reasoning + measure-first + memo restraint.

## Common Mistakes

- **Rendering the whole dataset** and hoping CSS `overflow:auto` saves you. It doesn't — the
  nodes still exist.
- **`memo`/`useCallback`/`useMemo` everywhere** "to be safe." Adds cost and complexity; often
  no-ops. Measure.
- **Positioning virtual rows with `top`/`margin`** → reflow every frame. Use `transform`.
- **Optimizing before measuring.** The bottleneck is usually elsewhere (often network or a
  layout thrash, not React).
- **Debouncing the wrong thing** (debouncing the render instead of the server call).

## Summary

- **Perf = do LESS / LATER / ELSEWHERE, and MEASURE first.** Every tool is one of those.
- **Virtualization:** render proportional to the viewport (~20 rows), recycle on scroll,
  `transform` (composited, Ch 07) not `top` (reflow). Tradeoffs: find/a11y/variable-height.
- **Memo family** trades compare-cost for skipped renders — worth it only for measured hot spots
  with stable props; prefer composition; never blanket-apply (Interviewer's trap).
- **Code-splitting/transitions/workers** are do-later/do-elsewhere; **CWV (LCP/INP/CLS)** grade
  the real-user result; the **Profiler/Performance panel** tell you where to look.
