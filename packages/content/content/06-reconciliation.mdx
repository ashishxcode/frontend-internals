## The one mental model

> **React does NOT compare your new tree against the DOM. It compares the new element tree
> against the previous one, position by position. At each position it asks ONE question: "same
> type as before?" Same type means keep the DOM node and the Fiber (and its state), just update
> props. Different type means throw away that node and its entire subtree (and its state) and build
> fresh. `key` overrides "position" with "identity" so React can match items that moved.**

From this you understand why changing a wrapper `<div>` to a `<section>` wipes child state. You see why list state gets cross-wired without keys. You can tell why index-as-key causes bugs on reorder or insert. And you know why a full tree diff would be too slow. No need to memorize "rules of keys."

## Key Mental Models

- **Diff is per-position, type-first.** Type mismatch means unmount subtree and mount new.
- **State is tied to position-in-tree (a Fiber), not to your JSX variable.** Move or replace the
  position and the state goes with the Fiber, or it dies.
- **`key` re-defines identity within a list**, so reorders reuse the right Fiber.

---

## Introduction

You have seen "add a key" warnings. You may have seen a bug where an input's value jumped to the wrong row. Both come from one thing: how React matches this render's elements to last render's Fibers. Understand this matching model and you can predict when state survives, resets, or moves.

---

## Problem

A general optimal tree diff is **O(n³)**, far too slow for UI that re-renders constantly. React needs "good enough" diffing in **O(n)**. It uses assumptions that hold for UIs.

Two heuristics make it linear:
1. **Different element types produce different trees.** If the type changed at a position, do not bother diffing children. Replace the whole subtree.
2. **Developers can mark stable identity with `key`.** Lists that reorder can be matched by key instead of position.

```mermaid
flowchart TD
  pos["compare element at this position: old vs new"] --> q{same type?}
  q -->|no| replace["unmount old subtree + state<br/>mount new subtree fresh"]
  q -->|yes| keep["reuse DOM node + Fiber + state<br/>update changed props, recurse to children"]
```

## Engine Simulation: type change wipes state

```jsx
{isEditing ? <input defaultValue={name} /> : <input defaultValue={name} />}  // same type
```
Same type at the same position. React reuses the input Fiber. Its internal state (cursor, uncontrolled
value) survives. Now:

```jsx
{isWide ? <div><Profile/></div> : <section><Profile/></section>}
```
`div` to `section` is a **type change** at that position. React unmounts the whole subtree.
`Profile` is destroyed and **remounted**, losing its state (and re-running its effects). Same
JSX `Profile`, but it sits under a position whose type flipped. State lives on the Fiber at that
position, and the position was rebuilt.

```
render A:  <div>      render B:  <section>
             └ Profile(state=X)     └ Profile(state= fresh!)   ← type changed → remount
```

## Engine Simulation: lists and the index-key bug

```jsx
{items.map((it, i) => <Row key={i} item={it} />)}   // ❌ index as key
```

You have rows A, B, C (keys 0,1,2). Each `<Row>` holds local state (say an open/closed toggle).
Now you **prepend** X. The list becomes X, A, B, C with keys 0,1,2,3:

```
before:  key0→A(stateA)  key1→B(stateB)  key2→C(stateC)
after :  key0→X          key1→A          key2→B          key3→C
match by key:
  key0: A→X  reuse Fiber → X now shows A's state!   (toggle leaks)
  key1: B→A  reuse Fiber → A shows B's state
  ...
```

React matched by `key=0` and decided "row 0 is the same row, just new props." So it kept row 0's
**state** and handed it to X. Every row's state shifted to the wrong item. The fix is a key that
follows the *item*, not the slot:

```jsx
{items.map((it) => <Row key={it.id} item={it} />)}   // ✅ stable identity
```

Now `key=A.id` matches A's Fiber wherever A moves. Prepending X just mounts one new Fiber and
reuses the rest correctly.

```mermaid
flowchart LR
  subgraph indexkey["key = index (bad)"]
    direction TB
    i["prepend X → every position's<br/>'identity' shifts → state mismatched"]
  end
  subgraph idkey["key = item.id (good)"]
    direction TB
    g["X gets a new Fiber;<br/>A,B,C keep theirs"]
  end
```

**When is index-as-key fine?** Only for static lists that never reorder, insert, or delete in the middle, and whose rows hold no state. Otherwise use a stable id.

## React Internals

- During `beginWork` (Ch 04), React reconciles a Fiber's children. For single children it does a type comparison. For arrays it builds a key-to-Fiber map from the current children. It matches new children by key, falling back to index when there is no key.
- Match with same type means clone the Fiber as WIP and update props. No match or type change means flag for deletion (old) and placement (new). These flags drive the commit phase DOM ops.
- The "reset state on prop change" trick: give a component a `key` that changes when you *want* a remount. For example, `<Editor key={docId} />` resets editor state per document. This uses keys deliberately to force identity change.

---

## Interview Discussion (reason first)

**Q1. "Why does React need keys?"**

*Plausible-but-wrong:* "For performance, so it renders faster."

*Correction:* Primarily for **correctness** of identity in dynamic lists. Performance is a bonus.
Without stable keys React matches by position. So reorders and inserts make it reuse the wrong
Fiber. State and DOM attach to the wrong item. Keys tell React "this element is *this* item,
wherever it moved."

*Model answer:* "Keys give list children stable identity so React matches by item, not slot.
Without them (or with index keys), inserting or reordering reuses Fibers for the wrong items.
This leaks state and causes subtle bugs."

**Q2. "Changing a wrapper from div to section reset my form. Why?"**

*Model answer:* "Type change at that position makes React unmount the subtree and mount fresh.
Descendant state and effects reset. State is tied to the Fiber at a position. The position was
rebuilt."

**Q3. "When is `key={index}` acceptable?"**

*Model answer:* "Static, never-reordered, stateless lists. Anything that reorders, inserts, deletes,
or whose rows hold state needs a stable id."

*Scoring:* full answer means identity-not-perf, type-change-remount, and index-key bug walk. Fail answer means "keys just silence the warning."

## Common Mistakes

- **Index keys on dynamic or stateful lists.** State leaks to wrong rows.
- **Using a random key** (`key={Math.random()}`). This remounts every render and destroys state and perf.
- **Forgetting that a type change remounts.** You lose animation or input state when restructuring.
- **Assuming React diffs against the DOM.** It diffs element tree against previous element tree.

## Summary

- React diffs **new element tree vs previous**, **per position**, **type-first**. It runs in O(n) via two heuristics: different type means different tree, and `key` marks identity.
- **Same type means reuse Fiber, DOM, and state, then update props. Different type means unmount subtree and remount.** State and effects reset.
- **State lives on the Fiber at a position**, not on your JSX. `key` redefines identity in lists so reorders match by item. Index and random keys break this and leak or destroy state.
- Use a changing `key` deliberately to force a reset.

# ═══ Internals Deep-Dive (source-verified) ═══

> Verified against `facebook/react` v19.2.0, `react-reconciler/src/ReactChildFiber.js`. The
> position/type/key model above is exactly what the code does. Here are the real functions.

## A. Two reconcilers from one body

`createChildReconciler(shouldTrackSideEffects)` produces two functions from the same code:
- **`mountChildFibers`** (`shouldTrackSideEffects = false`) handles initial mount. It skips deletion tracking.
- **`reconcileChildFibers`** (`true`) handles updates. It tracks `Placement` and `ChildDeletion` flags
  (Ch 04) that drive the commit's DOM ops.

`reconcileChildFibersImpl` branches on the new child: single element
(`$$typeof === REACT_ELEMENT_TYPE`) calls `reconcileSingleElement`; string/number calls text node; array calls
`reconcileChildrenArray`; iterable calls the iterator variant.

## B. Single element: the exact reuse rule

`reconcileSingleElement` / `updateElement`: reuse the existing fiber **iff `key` matches AND
`current.elementType === element.type`**. Then `useFiber(current, props)` (= `createWorkInProgress`,
Ch 04). **State is preserved**. Otherwise `deleteChild` + create a fresh fiber with `Placement`, so **state is lost**. (Note the compared field is **`elementType`**, not `type`. They differ only for lazy components and dev hot-reload.) This is the Ch 06 "same type means reuse, different type means remount" rule at the line level.

## C. List diff: the real two passes (`reconcileChildrenArray`)

React deliberately does **no** two-ended diff. Source comment: *"This algorithm can't optimize by searching from both ends since we don't have backpointers on fibers."* Two passes:

```mermaid
flowchart TD
  p1["Pass 1: walk old fibers & new children in lockstep (updateSlot)"] --> brk{key lines up?}
  brk -->|yes| reuse[reuse fiber, advance both] --> p1
  brk -->|no, break| p2["Pass 2: mapRemainingChildren → Map keyed by key (or index if keyless)"]
  p2 --> match["each remaining new child: updateFromMap (match anywhere by key)"]
  match --> del["leftover map entries → deleteChild"]
```

1. **Pass 1** loops old + new in index lockstep. It calls **`updateSlot`**, which returns `null` the moment a key does not line up. This causes a `break`. (If new children run out, delete the rest. If old run out, fast-path insert via `createChild`.)
2. **Pass 2** uses **`mapRemainingChildren`** to build a `Map` keyed by **`fiber.key`**, falling back to `fiber.index` when key is null. Then **`updateFromMap`** matches each remaining new child by `newChild.key ?? newIdx`. Leftover map entries are deleted.

**`placeChild(newFiber, lastPlacedIndex, newIndex)`** decides moves. If a matched fiber's old index is less than `lastPlacedIndex`, it marks `Placement` (a move). Otherwise it advances `lastPlacedIndex`.

## D. Why index keys corrupt state: the exact mechanism

Keyless (or index-keyed) children are stored under their **index**. `updateFromMap` looks up `newChild.key === null ? newIdx : newChild.key`. So index keys make React match **by position**. When you prepend, insert, or reorder, position N now holds a *different logical item*. But its key (the index) is unchanged. React reuses that position's fiber and hands it the new item. The old fiber's **state and uncontrolled DOM (input text, focus, checkbox) attach to the wrong row**. A stable `key={item.id}` makes the Map match the item wherever it moved. This is exactly the Ch 06 prepend bug in source terms. (`key={Math.random()}` is the opposite failure: every key is new every render so every fiber is deleted and remade. All state is lost and perf is worst.)
