## The one mental model

> **Server state is not your state. It is a CACHE of data that lives on someone else's machine
> and can go stale at any moment. So you do not "store" it in `useState`. You cache it keyed by
> *what you asked for* (the query key). You adopt a policy: show the cached copy instantly,
> and revalidate in the background (stale-while-revalidate). TanStack Query is that cache plus that
> policy engine. Every feature (staleTime, refetch, invalidation, optimistic updates) is just
> a knob on "how fresh must this cached copy be, and when do I re-ask?"**

From this you understand the whole API. You will not memorize options. You will ask "what does this knob do to cache freshness?" and the name explains itself.

## Key Mental Models

- **Server state = remote, shared, async, can-go-stale.** Client state = yours, synchronous.
- **Query key = cache key.** Same key means same cached entry, deduped across components.
- **stale-while-revalidate:** serve cache now, refetch in background, swap when fresh arrives.
- **`staleTime`** = how long "fresh" (no refetch). **`gcTime`** = how long an *unused* entry stays in memory before garbage collection. Different axes.

## Introduction

The classic mistake is treating fetched data like local state. You use `useEffect(fetch)` plus `useState(data)` plus manual loading and error flags. You duplicate this in every component. You have no caching, no deduping, and no refetch policy. TanStack Query exists because "data from a server" has properties that local state does not. It is shared. It goes stale. Multiple components want the same data. You need loading, error, and refetch handling. Once you see fetched data as a *cache*, the library becomes obvious.

## Problem

```jsx
function Contacts() {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);
  useEffect(() => {
    setLoading(true);
    fetch("/contacts").then(r => r.json()).then(setData).catch(setError)
      .finally(() => setLoading(false));
  }, []);
  // ...repeat this in every component that needs contacts
}
```

Problems: no caching (refetches on every mount), no dedup (two components means two requests), no background refresh, no shared invalidation, stale-closure risk (the effect callback captures an outdated variable when deps are incomplete, see Ch 05), and boilerplate everywhere. The root issue: you are hand-rolling a cache badly. The fix: **extract the cache and the fetch policy into a library**, keyed by the request.

```mermaid
sequenceDiagram

  participant Q as Query Cache
  participant S as Server
  C->>Q: useQuery(['contacts', filters])
  alt cache miss or stale
    Q->>S: fetch
    S-->>Q: data → store under key
  end
  Q-->>C: cached data (instant if present)
  Note over Q: if stale, refetch in background, then push fresh data
```

## Engine Simulation: the cache lifecycle

```jsx
const { data, isLoading, isFetching } = useQuery({
  queryKey: ['contacts', { filters, page }],
  queryFn: () => api.getContacts(filters, page),
  staleTime: 30_000,   // fresh for 30s, no refetch on remount or focus within 30s
  gcTime: 5*60_000,    // keep unused cache 5 min, then drop
});
```

Trace it:

```
mount #1:  key ['contacts',{...}] not in cache → fetch → store → status 'success'
           isLoading true→false; data shown
remount within 30s:  key present & FRESH (within staleTime) → serve cache, NO fetch
remount after 30s:   key present but STALE → serve cache instantly + refetch in background
                     (isFetching=true while data still shows; swaps when fresh arrives)
component unmounts:  entry becomes 'inactive'; gcTime timer starts
no observers for 5min: entry garbage-collected
```

Two distinct timers, the classic interview confusion:
- **`staleTime`** controls *refetching* (freshness). High staleTime means fewer requests.
- **`gcTime`** controls *memory* (how long an unused entry survives so a remount is instant).

`isLoading` = first load, no data yet. `isFetching` = any fetch in flight (including background revalidation while showing stale data). Show a subtle spinner on `isFetching` but content on `data`. This is the stale-while-revalidate UX.

## Mutations and optimistic updates (the contacts status flip)

Interviewer's table edits a contact's status inline. You do not want to refetch 500k rows. Two moves:

**Invalidate** (simple): after the mutation, mark queries stale so they refetch.
```jsx
const m = useMutation({
  mutationFn: updateStatus,
  onSuccess: () => qc.invalidateQueries({ queryKey: ['contacts'] }),
});
```

**Optimistic update** (instant UX): patch the cache *before* the server confirms, roll back on error. This is what makes a 500k-row table feel instant.
```jsx
useMutation({
  mutationFn: updateStatus,
  onMutate: async (next) => {
    await qc.cancelQueries({ queryKey: ['contacts'] });      // stop races
    const prev = qc.getQueryData(['contacts']);              // snapshot for rollback
    qc.setQueryData(['contacts'], patch(prev, next));        // patch cache now
    return { prev };
  },
  onError: (_e, _next, ctx) => qc.setQueryData(['contacts'], ctx.prev), // rollback
  onSettled: () => qc.invalidateQueries({ queryKey: ['contacts'] }),    // reconcile
});
```

```mermaid
stateDiagram-v2
  [*] --> idle
  idle --> optimistic: onMutate (patch cache, snapshot)
  optimistic --> confirmed: onSuccess (server agrees)
  optimistic --> rolledback: onError (restore snapshot)
  confirmed --> idle: onSettled (invalidate→reconcile)
  rolledback --> idle
```

## Infinite or windowed contacts list

`useInfiniteQuery` pages data with a cursor. Pair it with virtualization (renders only visible rows in DOM for performance, see Ch 08). The **query cache** holds the fetched pages (data layer). The **virtualizer** decides which cached rows are in the DOM (view layer). Real-time status events call `setQueryData` to patch a row in the cache. If the row is visible it re-renders in place (React uses `key` to match elements across renders for efficient in-place updates, see Ch 06). If off-screen, it is correct when scrolled to. This is the clean "data layer vs DOM layer" separation that Interviewer probed. See the Interviewer's real-time contacts-table worked example (guide §2f).

## Interview Discussion (reason first)

**Q1. "Why not just `useState` + `useEffect` to fetch?"**
> "Because server data is a *cache*, not local state. It is shared, async, and goes stale. Hand-rolling means no dedup, no caching, no background refresh, no shared invalidation, and stale-closure bugs. TanStack centralizes the cache keyed by the request and gives a refetch policy."

**Q2. "staleTime vs gcTime?"**
> "`staleTime` = how long data is considered fresh, so no refetch happens (controls requests). `gcTime` = how long an *unused* cache entry stays in memory before it is dropped (controls memory and instant remounts). They are orthogonal. A query can be stale but still cached."

**Q3. "Edit one contact's status in a 500k table, how, without refetching everything?"**
> "Optimistic update: `onMutate` cancels in-flight queries, snapshots, and calls `setQueryData` to patch just that row. The UI flips instantly. `onError` rolls back to the snapshot. `onSettled` invalidates to reconcile with the server. No full refetch."

*Scoring:* full answer includes cache framing, two-timers, and optimistic flow with rollback. Fail answer says "it is like Redux for API calls."

## Common Mistakes

- **Putting server data in Redux or `useState`** and syncing manually. You are re-implementing TanStack badly.
- **Confusing `staleTime` and `gcTime`** (or thinking `staleTime: Infinity` frees memory. It does not. That is `gcTime`).
- **Unstable query keys** (new object identity that is actually the same query, or missing a variable the query depends on, which leads to stale data).
- **Optimistic update without `cancelQueries`.** An in-flight refetch overwrites your patch.
- **Forgetting rollback** on error. The UI then lies.

## Summary

- **Server state is a cache of remote data**, not local state. That single reframe justifies the whole library.
- **Query key = cache identity** (dedup and sharing). **stale-while-revalidate**: serve cache, refetch in background.
- **`staleTime`** controls freshness and refetching. **`gcTime`** controls how long unused cache survives in memory. Different axes.
- **Mutations**: `invalidateQueries` (simple) or **optimistic update** (`onMutate` patch and snapshot, then `onError` rollback, then `onSettled` reconcile) for instant UX on big tables.
- Pair `useInfiniteQuery` (data layer) with virtualization (view layer) for clean separation.
