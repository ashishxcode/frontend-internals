## The Question Everyone Gets Wrong

"How do you manage state in React?"

If you answer with a library name — Redux, Context, Zustand — you just walked into a trap. The real question is: **state has kinds**, and each kind needs a different tool.

Here is what happens when teams get this wrong: they pick Redux for everything. Every button click dispatches an action. A simple toggle goes through a reducer. Then someone puts server data in Redux. They reimplement caching, deduplication, and background refetching — badly. TanStack Query already does all of that.

## The Mental Model

Two questions tell you which tool to use:

1. **Who needs this?** One component? A few nearby? Everyone? The server?
2. **How often does it change?** Never? Sometimes? Every frame?

**Analogy:** A filing cabinet with four drawers. "My Stuff" (one component) — use `useState`. "Shared Stuff" (a few components) — lift state or use Zustand. "Company-Wide Stuff" (everyone) — Context if it rarely changes, Zustand if it changes often. "Other People's Stuff" (server data) — TanStack Query, not a client store at all.

The core insight: **place state on the scope × volatility grid and the right tool is obvious.**

| Drawer | Scope | Volatility | Tool |
|--------|-------|------------|------|
| My Stuff | One component | Any | `useState` |
| Complex Transitions | One component, many fields | Any | `useReducer` |
| Company-Wide (rare) | Everyone | Changes rarely | Context |
| Company-Wide (frequent) | Everyone or selective | Changes often | Zustand |
| Big Team + Middleware | Everyone | Complex flows | Redux Toolkit |
| Other People's Stuff | Server | Varies | TanStack Query |

## The Context Problem

Context is a megaphone — it broadcasts to everyone. No way to whisper to just the components that need the changed value.

```jsx
const AppCtx = createContext();
function Provider({ children }) {
  const [user, setUser] = useState();
  const [theme, setTheme] = useState("light");
  const [cartCount, setCartCount] = useState(0); // changes often
  return (
    <AppCtx.Provider value={{ user, theme, cartCount, setTheme, setCartCount }}>
      {children}
    </AppCtx.Provider>
  );
}
```

Every time `cartCount` changes, the `value` object is a new reference. React re-renders **every** component that calls `useContext(AppCtx)` — even ones that only read `theme`. This is fine for rarely-changing globals. For anything volatile, Context kills performance.

## Zustand: Email, Not Group Chat

Zustand lives outside React. Components subscribe with selectors — functions that pick exactly the slice they need.

```js
const useStore = create((set) => ({
  count: 0,
  user: null,
  inc: () => set((s) => ({ count: s.count + 1 })),
}));

function Counter() {
  const count = useStore((s) => s.count);
  return <button onClick={useStore.getState().inc}>{count}</button>;
}
function Profile() {
  const user = useStore((s) => s.user);
  return <span>{user?.name}</span>; // does NOT re-render when count changes
}
```

When `inc()` runs, Zustand checks every subscriber's selector output. If the selector returned the same value (by `Object.is`), the component does not re-render. `Profile` picks `s.user` — `user` did not change, so `Profile` is skipped.

**The selector trap:** `useStore((s) => ({ a: s.a, b: s.b }))` creates a new object every call, defeating `Object.is`. Fix: split into two selectors or use Zustand's `shallow` comparator.

## useReducer: Centralized Transitions

When a component has complex state with many interdependent transitions, `useReducer` bundles them into one pure function:

```js
function reducer(state, action) {
  switch (action.type) {
    case "field":   return { ...state, [action.name]: action.value };
    case "submit":  return { ...state, status: "submitting", error: null };
    case "error":   return { ...state, status: "idle", error: action.error };
    default:        return state;
  }
}
```

The reducer is a pure function — given state and action, it returns the next state. Testable without rendering anything. Scales conceptually to Redux — a Redux reducer is the same pattern at app scale.

## Server State: Don't Put It in a Client Store

When you fetch data from an API and store it in Redux or Zustand, you now need caching, deduplication, background refetching, garbage collection, and optimistic updates. TanStack Query handles all of this.

```js
const { data, isLoading, isFetching } = useQuery({
  queryKey: ['contacts', { filters, page }],
  queryFn: () => api.getContacts(filters, page),
  staleTime: 30_000,
  gcTime: 5 * 60_000,
});
```

Server state is not your state. It is a cache of someone else's data. Treat it that way.

**Modern default:** TanStack Query (server) + useState (local) + a little Zustand (global client). Redux only when you need its middleware ecosystem at scale.

## Common Mistakes

- One giant Context for everything that changes often — causes app-wide re-renders.
- Server data in a client store — you reimplement TanStack Query badly.
- Redux for trivial state — a toggle does not need actions, reducers, and middleware.
- Selectors returning new objects — defeats reference equality optimization.
- Lifting state too high "just in case" — causes unnecessary re-renders.

## Q&A

**Q: Theme, contacts list, a modal flag, cart count across the app — which tool for each?**
Theme: Context (rarely changes, broadcast cost negligible). Contacts list: TanStack Query (server state, handles caching). Modal flag: useState (local to one component). Cart count across app: Zustand (many components read selectively, frequent changes).

**Q: Why does one giant Context re-render everything?**
Context broadcasts to every consumer on every value change. A new `value` object reference (from any state update in the provider) triggers re-renders in all consumers — even those reading unchanged values. Split contexts by volatility or move frequent state to Zustand.

**Q: A component reads `useStore((s) => ({ a: s.a, b: s.b }))` and re-renders too often. Why?**
The selector creates a new object every call. Zustand uses `Object.is` — new reference means re-render every time. Fix: split into two selectors returning primitives, or use Zustand's `shallow` comparator.

**Q: Why is useReducer more testable than four useState calls?**
The reducer is a pure function. Given state and action, it returns the next state deterministically. Test without rendering: `expect(reducer(initial, action)).toEqual(expected)`. With separate useState, testing requires rendering the component and triggering events.
