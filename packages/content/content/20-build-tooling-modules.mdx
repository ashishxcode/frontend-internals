## The 70KB Library You Never Asked For

You import `_ from "lodash"` and ship the entire 70KB library. You use one function. Your bundle is huge. The dev server is slow. These are real questions — the job description names Vite, and the whole area becomes small once you see the bundler as a graph optimizer whose job is shipping less code.

The mistake that wastes kilobytes on every deploy: CJS `require` is dynamic. The bundler cannot statically know which exports you use, so it conservatively includes everything. ESM imports are static and analyzable — the bundler can trace exactly what you need and drop the rest.

## The Mental Model

A bundler is a **graph optimizer**. It starts at your entry file, follows every `import` to build a dependency graph, then emits the smallest, fastest set of files the browser can load. Two jobs: combine modules so the browser makes few requests, and ship less code via tree-shaking, code-splitting, and minification.

**Analogy:** CJS is a restaurant that preps every dish on the menu every morning, even though most will never be ordered. ESM is ordering — the kitchen only prepares what you ask for.

The core insight: **ESM is static and analyzable. CJS is dynamic and opaque. That one difference determines everything about what a bundler can optimize.**

## ESM vs CJS

```js
// ESM: static, analyzable — tree-shaking works
import { debounce } from "lodash-es";

// CJS: dynamic, opaque — bundler includes everything
import _ from "lodash";
```

CJS `require()` can appear conditionally at runtime. The bundler conservatively includes both branches. ESM `import` statements are at the top level, static, and analyzable. The bundler traces exactly which named exports are used and eliminates everything else.

Result: `lodash-es` with named imports shrinks from 70KB to ~4KB for `debounce` plus its helpers.

## How Vite Works

Vite uses two modes:

**Dev (esbuild):** Written in Go. Transpiles TypeScript and JSX on demand. Does NOT bundle your code. The browser imports modules via native ESM. Instant startup. HMR only re-transpiles the changed file.

**Build (Rollup):** Bundles everything. Tree-shakes, code-splits, minifies. Output is optimized files for the network.

```js
// Dev: browser does native ESM imports, esbuild transpiles on-demand
// Build: Rollup bundles into dist/assets/index-abc123.js
```

## Tree-Shaking

Dead code elimination at the module boundary. Works because ESM has static structure — the bundler knows the full import/export graph before code runs.

Trace what happens with `import { debounce } from "lodash-es"`:

1. Bundler resolves lodash-es, parses its exports.
2. Marks `debounce` as used, everything else as unused.
3. Checks unused exports for side effects — if a top-level statement runs code on import, it must be kept.
4. Walks deeper into only `debounce`'s dependencies.
5. Eliminates everything else. Output: ~4KB instead of 70KB.

Side effects block tree-shaking. If a module has `"sideEffects": true`, the bundler includes everything. You can override with `"sideEffects": false` in your package.json, but verify — marking a CSS-in-JS library as side-effect-free breaks styles.

## Code-Splitting

Route-level splitting defers rarely-used code to lazy chunks:

```tsx
const Chart = React.lazy(() => import("./Chart"));

function Dashboard() {
  return (
    <Suspense fallback={<div>Loading chart...</div>}>
      <Chart data={data} />
    </Suspense>
  );
}
```

Rollup sees `import("./Chart")` and creates a separate chunk containing Chart and its dependencies (200KB). The initial bundle no longer includes chart code. The chunk downloads only when the user navigates to that route.

Before: initial bundle 340KB, load time 3.2s on 3G. After: 120KB, 1.1s. No architecture change — just named ESM imports and route-level code-splitting.

## Common Mistakes

- `import _ from "lodash"` instead of `lodash-es` named imports — defeats tree-shaking.
- No code-splitting — one giant entry bundle blocks first paint.
- Assuming dev performance equals prod — dev is unbundled, prod is bundled.
- Shipping source maps publicly — upload to error tracker only.

## Q&A

**Q: Walk through what a bundler does from entry to output.**
Parse the entry file into an AST. Resolve every import (file paths, node_modules). Build the dependency graph recursively. Transform each module (strip TypeScript, compile JSX). Tree-shake unused exports. Code-split dynamic `import()` expressions into separate chunks. Combine remaining modules. Minify. Emit files with content hashes for cache busting.

**Q: A teammate imports `_ from "lodash"`. Why does this break tree-shaking?**
Lodash ships CJS by default. CJS `module.exports` is a dynamic object — the bundler cannot statically determine which properties are used. Switch to `lodash-es` with named imports: `import { debounce } from "lodash-es"`. The bundler traces the named export and drops everything else.

**Q: Vite is fast for small apps. Does it stay fast at 1000 modules?**
Dev uses native ESM — every module is a separate HTTP request. Under ~500 modules this is near-instant. Beyond that, the HTTP request waterfall grows (100+ module requests before render). HMR stays fast regardless. Production builds (Rollup) are not affected — everything is bundled.

**Q: A library has `"sideEffects": true` but you know it has no side effects. What happens?**
Tree-shaking is disabled for that package — every module is included regardless of what you import. Override with `"sideEffects": false` in your package.json for that package. Verify in production mode — if the library actually injects styles or polyfills on import, marking it side-effect-free breaks things.
