## Why This Exists

The CEO clicks "Export to CSV." Nothing happens for two seconds. The React dev blames the API, the backend dev blames React, and ops says the server is fine. Nobody can trace that click from button to screen.

That's the real problem. Most of us learn JavaScript from one blog, React from a course, and browser rendering from a YouTube video. Each makes sense alone. But when you need to answer "why did setState not update immediately?" — you freeze. You know the words. You don't have a model.

This roadmap fills those gaps in dependency order, so each layer builds on the one before it.

## The Dependency Graph

```
Layer 1:  JavaScript Runtime          (ch 01-02)
    ↓
Layer 2:  React Internals             (ch 03-06)
    ↓
Layer 3:  Browser Internals           (ch 07)
    ↓
Layer 4:  Performance                 (ch 08)
```

You cannot skip layers. Part 2 assumes Part 1. Part 3 assumes Part 2. If you try Part 6 without Part 2, you'll read about TanStack Query caching without understanding why React re-renders. You'll get the words but not the understanding.

Each layer answers a question the previous layer raises:
- JavaScript runtime explains *why* closures and the event loop exist
- React internals explains *how* setState becomes a DOM update
- Browser rendering explains *what* the browser does with those DOM changes
- Performance explains *how* to measure and fix bottlenecks

## How to Use This

Three rules:

**Read in order.** Open chapter 01. Read it. When you finish, open chapter 02. This isn't a buffet — it's a curriculum. Trust the order.

**Trace through the stack.** When you learn about fiber lanes in chapter 04, connect it back to the event loop from chapter 02. Ask: "how does the scheduler use microtasks to yield?" If you can't answer, re-read chapter 02. Passive reading doesn't work. You have to stitch the layers together.

**Explain out loud.** After each chapter, close the file and explain the concept to an imaginary interviewer. If you stumble, you didn't internalize it. Re-read.

## What You'll Build

By the end, you can trace any user interaction through the entire stack. When an interviewer asks "what happens when a user clicks a button?", you'll describe the JS engine executing the handler, the event loop draining microtasks, React batching the setState calls, Fiber building the work-in-progress tree, hooks reading from state slots, reconciliation diffing the element tree, the browser running layout and paint, and performance tools measuring the interaction latency.

Not the definition. The trace.

## Common Mistakes

**Reading without simulating.** You can read the event loop chapter in 30 minutes. Understanding it takes longer. Close the file and trace code through your mental model until it matches what the browser actually does.

**Skipping what you think you know.** You use closures every day. Chapter 01 covers them anyway. The moment you think you know something is usually when you've missed the deepest part of it.

**Not connecting the layers.** After chapter 02, every time you see asynchronous code, ask yourself what the event loop is doing. After chapter 04, every time you use setState, trace the fiber scheduling. This active connection is what builds the mental model.
