## Problem

I spent 6 months preparing for my first SDE-2 interview. I read 40 blog posts, watched 12 hours of video, and built 3 projects. When the interviewer asked "how does React batch state updates?" I could not connect closures to the event loop to the scheduler. I knew each piece in isolation. I had no mental model connecting them.

This is the real problem. The web platform is a stack of interdependent systems. JavaScript powers React. React powers the browser. The browser paints what React produced. If you learn them as separate topics, you will have gaps. Interviewers probe gaps. They ask "why" until you hit a missing foundation.

This roadmap exists to fill those gaps in dependency order.

## Why This Roadmap Exists

Most interview resources are written by content marketers or course creators. They list APIs. They summarize features. They do not show you how the JavaScript engine, React, and the browser collaborate on a single button click.

This handbook is different. It is written by an engineer who has been through the interview process, mentored other engineers through it, and noticed the same gaps appearing every time. The structure mirrors how the platform actually works: bottom up.

You start at the JavaScript runtime because React runs on JavaScript. You learn React internals because the browser runs React. You learn browser rendering because the user perceives performance through paint. Each layer answers a question the previous layer raises.

If you have ever asked "why does this work?" and got an unsatisfying answer, this handbook is for you.

## How It Helps You

Three concrete ways this roadmap changes how you prepare:

**1. It prevents wasted time.**
You will not study React fiber before understanding closures. You will not memorize caching headers before understanding the event loop. Every chapter assumes only what came before it. You read in order and nothing surprises you.

**2. It gives you a mental simulation engine.**
After Part 1, you can simulate what the JavaScript engine does when your code runs. After Part 2, you can simulate what React does when you call setState. After Part 3, you can simulate what the browser does when the DOM changes. By Part 10, you can trace a click through all of them.

**3. It prepares you for depth questions.**
Interviewers at SDE-2 level do not ask "what is a closure." They ask "what happens when this closure captures a variable from a loop and you call it after the loop finishes?" The roadmap builds the precise understanding needed for those questions.

## How to Use This Handbook

Three rules:

**Read in order.** Open chapter 01. Read it. Do the simulations in your head. When you finish, open chapter 02. Do not jump.

**Trace every concept through the stack.** When you learn about fiber lanes in chapter 04, connect it back to the event loop from chapter 02. Ask: "how does the scheduler use microtasks to yield to the browser?" If you cannot answer, re-read the relevant part of chapter 02.

**Explain out loud.** After each chapter, close the file and explain the concept to an imaginary interviewer. If you stumble, you did not internalize it yet. Re-read.

## What You Will Build

By the end of this handbook, you will be able to:

- Explain what happens between a user clicking a button and the screen updating. Every step. Every system involved.

- Implement common JavaScript utilities from scratch (debounce, throttle, deep clone, Promise.all) and explain how each uses closures, the event loop, or value vs reference.

- Design a component API with proper tradeoffs between controlled and uncontrolled, composition and configuration.

- Debug a production performance issue by tracing from the user complaint through the React DevTools profiler to the browser performance panel.

- Answer SDE-2 interview questions with depth, not buzzwords. When asked "what is reconciliation," you will explain the problem React solved, the O(n) heuristics, the key algorithm, and the tradeoffs. Not the definition.

## Common Mistakes Engineers Make With This Material

**Reading without simulating.** You can read the event loop chapter in 30 minutes. Understanding it takes longer. You must close the file and trace code through your mental model until it matches what the browser actually does.

**Skipping what you think you know.** You already use closures every day. Chapter 01 covers them anyway. Read it. I guarantee there is something you did not know about how the engine stores them.

**Not connecting the layers.** After chapter 02, every time you see asynchronous code, ask yourself what the event loop is doing. After chapter 04, every time you use setState, trace the fiber scheduling. This active connection is what builds the mental model.

## The Teaching Pattern

Every chapter follows 13 sections. Once you know the pattern, you can extract knowledge from any chapter quickly:

```
Problem          →  What pain does this solve?
Why Failed       →  What did people do before and why was it bad?
Mental Model     →  One sentence framing.
Visualization    →  Diagram or flow.
Engine Sim       →  Step by step trace through the runtime.
Implementation   →  How it works internally.
Real World       →  Production example.
Tradeoffs        →  What you gain and lose.
Mistakes         →  What beginners get wrong.
Interview Answer →  Mid-level / Senior / Lead variants.
Follow-up Qs     →  5 progressively harder questions.
Mental Trigger   →  One phrase for instant recall.
Revision         →  Bullet summary.
```

## Follow-up Questions

**Q1: If you had to remove 3 chapters from this roadmap, which would you remove and why?**

Ch 09 (TypeScript) is a standalone skill. You can be a strong frontend engineer without deep TypeScript. Ch 27 (Git/CI/CD) is operational knowledge learned on the job. Ch 31 (Tailwind CSS) is a tool choice. Removing these lets you focus more time on core platform knowledge.

**Q2: Where would you place a chapter on Web Workers?**

Between Part 3 (Browser Internals) and Part 4 (Performance). Workers are a browser API that directly affects performance. The reader needs to understand the browser's thread model before they can understand why offloading work to a worker is valuable.

**Q3: Does the learning order change for a backend engineer switching to frontend?**

Yes. They need more time on Part 1 (JavaScript Runtime) because backend engineers often use JavaScript differently (Node.js patterns differ from browser patterns). They can skip Part 5 (TypeScript) if they already know typed languages. They should spend extra time on Part 3 (Browser Rendering) because the browser environment is new to them.

**Q4: How would you measure whether someone actually learned from this roadmap?**

Ask them to trace a click through all 10 layers without notes. If they can describe what happens in the JavaScript engine, then React, then the browser, then how performance tools measure it, then how networking affected the data, they internalized it. If they give isolated answers for each topic, they memorized but did not connect.

**Q5: What specific gaps appear if someone skips Part 1 and goes straight to React?**

They cannot explain closures, so they create stale closures in useEffect dependencies. They do not understand the event loop, so they cannot explain why setState does not update synchronously. They do not understand value vs reference, so they mutate state directly. They memorize React rules instead of deriving them from how JavaScript works.
