## The Problem That Hooks You

System design interviews are where SDE-2 candidates fail most. You can code well. Then the interviewer says "Design a component library" and you freeze. You start writing `useState` and `useEffect`. The interviewer looks bored.

This isn't a coding problem — it's a thinking problem. You needed to clarify requirements, define tradeoffs, and make architectural decisions. Instead, you jumped to implementation.

## The One Insight

**System design = making decisions that contain future change.** A zoning decision today prevents a factory from appearing next to school tomorrow. A data flow decision prevents a spaghetti-mess refactor in 6 months.

Think of it like building a city:
- **Zoning (Architecture)** — where things live: components, services, shared code.
- **Roads (Data Flow)** — how information moves: props down, events up, global state.
- **Utilities (Shared Services)** — build once, consume everywhere: auth, analytics, theming.
- **Building Codes (Standards)** — accessibility, testing, documentation rules.

## The Framework: 5 Steps

### Step 1: Clarify Requirements

Before drawing anything, ask questions. Separate functional from non-functional.

```text
Functional:
- Button, Input, Modal, Table, Select components
- Theming support (light/dark)
- TypeScript-first

Non-functional:
- Tree-shakeable (import only what you use)
- Accessible (WCAG 2.1 AA)
- 50+ developers using it
- Versioned with semver
```

### Step 2: Define the API Contract

Before designing internals, define the external interface. This is where most candidates skip ahead.

```jsx
// Component Library
<Button variant="primary" size="md" loading={false}>Save</Button>
<Select options={countries} value={selected} onChange={setSelected} />

// Notification System
useNotifications() // returns { notifications, unreadCount, markRead, clearAll }
```

### Step 3: Design the Module Structure

```text
Component Library:
├── tokens/       (colors, spacing, typography)
├── primitives/   (Button, Input, Select)
├── composites/   (DataTable, Form, DatePicker)
├── patterns/     (auth flows, search, CRUD)
└── utils/        (cn(), useControllableState)
```

### Step 4: Choose State Management

Local state + context + URL state. Avoid global state unless you have a clear reason. Component library needs almost no global state. Notification system benefits from global context.

### Step 5: Handle Edge Cases

Loading (skeletons, not spinners), error (specific messages + retry), empty (message + CTA), offline (cache + indicator), accessibility (focus, keyboard, screen reader).

## Problem 1: Component Library / Design System

### Token System

```js
export const colors = {
  blue: { 500: '#3b82f6', 600: '#2563eb' },
  semantic: { success: '{colors.green.500}', error: '{colors.red.500}' },
};
```

Tokens solve "why does this button look different in that app?" — every visual decision is centralized.

### Component API Design

```jsx
// BAD: Too many props
<Alert type="error" title="Delete failed" message="..." showIcon closable onClose={handleClose} action={<Button>Retry</Button>} />

// GOOD: Compound components
<Alert variant="error" onClose={handleClose}>
  <Alert.Icon name="error" />
  <Alert.Title>Delete failed</Alert.Title>
  <Alert.Description>Could not delete the item</Alert.Description>
  <Alert.Action><Button onClick={retry}>Retry</Button></Alert.Action>
</Alert>
```

Compound components give consumers control over structure without growing the prop surface.

### Versioning

```text
v1.5: Add deprecation warning for old API
v1.8: Log console.warn when old API is used
v2.0: Remove old API (major version bump)
Codemods: jscodeshift transform rewrites old → new automatically
```

## Problem 2: Real-Time Notification System

### Transport: SSE over WebSocket

For notifications, SSE is usually right. You need server→client push, not bidirectional. WebSocket is overkill.

```text
Polling:   Simple, wasteful, delay up to interval.
SSE:       Simple, auto-reconnect, server→client push.
WebSocket: Full duplex, complex connection management.
```

### Connection Management

```jsx
class NotificationConnection {
  connect() {
    this.eventSource = new EventSource(this.url);
    this.eventSource.onopen = () => { this.retryCount = 0; };
    this.eventSource.onmessage = (event) => this.onNotification(JSON.parse(event.data));
    this.eventSource.onerror = () => {
      this.eventSource.close();
      this.scheduleReconnect(); // exponential backoff with jitter
    };
  }
}
```

Exponential backoff prevents hammering the server. Jitter prevents thundering herd.

### Delivery Guarantees

At-least-once delivery. Server assigns monotonically increasing IDs. Client tracks last seen ID (localStorage). On reconnect: "give me everything since ID X." Client deduplicates by ID.

## Problem 3: Data Table with Server-Side Operations

### Pagination

```text
Offset-based:  GET /api/users?page=3&limit=20
  Pros: Simple, jump to page N. Cons: Inconsistent when data changes.

Cursor-based:  GET /api/users?cursor=eyJpZCI6MTAwfQ&limit=20
  Pros: Consistent, efficient for large datasets. Cons: Can't jump to page N.
```

### Sorting/Filtering as URL State

```jsx
const [searchParams, setSearchParams] = useSearchParams();
// Sorting survives page refresh, is shareable, bookmarkable.
```

### Virtualization

For 1000+ rows, render only visible rows. Use `@tanstack/react-virtual` with `estimateSize` and `overscan`.

## Problem 4: Micro-Frontends

### When to Use

```text
Use when:    30+ engineers, distinct domains, deployment bottleneck.
Don't use:   Small team, single domain, new project.
```

### Module Federation

```js
new ModuleFederationPlugin({
  remotes: { payments: 'payments@https://payments.example.com/remoteEntry.js' },
  shared: { react: { singleton: true, requiredVersion: '^18.0.0' } },
});
```

`singleton: true` ensures only one React loads. Without it, multiple instances cause hook failures.

## Tradeoffs

| Decision | Gain | Cost |
|----------|------|------|
| Compound components | Extensible, less props | More setup code |
| SSE over WebSocket | Simpler, auto-reconnect | Server→client only |
| Cursor pagination | Consistent results | Can't jump to page N |
| Virtualization | Handles 100k+ rows | Complex scroll logic |
| Micro-frontends | Deployment independence | Build complexity |

The most important tradeoff: ease of initial implementation vs ease of future change.

## Common Mistakes

- Jumping to implementation — start with requirements, not code.
- Ignoring non-functional requirements — performance, accessibility, offline.
- One-size-fits-all solutions — "Use Redux" for everything.
- Forgetting edge cases — loading, error, empty, offline.
- Over-engineering from the start — micro-frontends for 5 people.
- No versioning strategy — how do you evolve the API without breaking consumers?

## Follow-up Questions

**Q1: How would you handle theming in a component library supporting both React and Vue?**
CSS custom properties as the theming layer — framework-agnostic, resolves at runtime. Both React and Vue reference `var(--color-primary)`. Framework-specific part is the theme provider that sets `data-theme` on root.

**Q2: Your notification system needs to work offline. How do you synchronize missed notifications?**
Three layers: IndexedDB for persistent storage, monotonically increasing IDs, server-side catch-up API (`GET /api/notifications?since={lastSeenId}`). Deduplicate by ID before adding to state.

**Q3: Two micro-frontends share a user session. How do you handle auth without coupling?**
Auth shell pattern: host app owns authentication. Sets HttpOnly cookie with JWT. Micro-frontends validate JWT independently (no network call). Host handles refresh. Micro-frontends never manage refresh tokens.

**Q4: A breaking change in Button affects 40 apps. How do you manage at scale?**
Deprecation lifecycle with codemods: add console.warn, publish jscodeshift transform, write migration guide, run visual regression testing, notify all consuming apps.
