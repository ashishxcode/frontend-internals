## Why This Matters

You rename a variable. Twenty tests break. You swap a `<div>` for a `<section>`. Ten more break. None caught a real bug — they just made you afraid to touch the code. The team stops refactoring. Shipping slows down.

The problem isn't that you're testing too much. It's that you're testing the wrong things.

## The Core Idea

**Test the behavior at a boundary, not the machinery underneath.**

Think of a vending machine. You don't open the back panel to check if gears turn. You put in a dollar, press B3, and expect a soda. That's the boundary. If the soda drops, someone can rewrite the entire mechanism and your test still passes.

For a React component, the boundary is what the user sees and does. Tests coupled to internals — state names, CSS classes, render counts — break on every refactor and test nothing real.

## The Testing Trophy

```
        /\        e2e (Playwright): real browser, critical flows only.
       /  \       High confidence, slow. Few.
      /----\
     /      \     INTEGRATION (RTL + MSW): components + interactions + mocked network.
    /        \    Best confidence per cost. The bulk of UI tests.
   /----------\
  /            \  unit: pure functions, hooks, utils. Fast. Many, for logic.
 /--------------\
  static: TypeScript and ESLint (catch whole class of bugs for free)
```

The testing trophy inverts the old pyramid. For UI code, integration tests give the most confidence per dollar. Unit tests don't catch UI bugs. E2E is too slow for everything.

## React Testing Library: Query Priority

RTL queries match how users interact with elements. The priority order:

```jsx
// 1. Role — tests accessibility
screen.getByRole("button", { name: /submit/i });

// 2. Label — tests form accessibility
screen.getByLabelText(/email/i);

// 3. Text — tests visible content
screen.getByText(/saved successfully/i);

// 4. Test ID — last resort, tests nothing about UX
screen.getByTestId("submit-button");
```

Role-first forces you to build accessible components. Each query tests a different contract with the user.

## userEvent vs fireEvent

```jsx
// ❌ fireEvent: single synthetic event, misses focus management
fireEvent.change(input, { target: { value: "hello" } });

// ✅ userEvent: full browser sequence, handles focus, disabled states
await userEvent.type(input, "hello");
```

`userEvent` fires `keyDown`, `keyPress`, `input`, `keyUp` for each character — just like a real keyboard. Always prefer it.

## Async Queries: getBy vs findBy vs queryBy

- **`getBy`** — sync, throws if not found. Use for elements present on render.
- **`findBy`** — async, waits for element to appear (polls every 50ms, up to 1000ms). Use for data fetched after mount.
- **`queryBy`** — sync, returns null instead of throwing. Use for asserting absence.

Using `getBy` for async content throws before data arrives. Using `findBy` for sync content works but adds unnecessary overhead.

## MSW: Mock the Network, Not Your Code

MSW intercepts at the network layer. The real `fetch` runs. Your real API client runs. Only the response is controlled.

```jsx
const server = setupServer(
  http.get("/contacts", () => HttpResponse.json([{ id: 1, name: "Ada" }]))
);

test("renders contacts from the API", async () => {
  render(<Contacts />);
  expect(await screen.findByText("Ada")).toBeInTheDocument();
});

test("shows error when API fails", async () => {
  server.use(http.get("/contacts", () => HttpResponse.error()));
  render(<Contacts />);
  expect(await screen.findByText(/error/i)).toBeInTheDocument();
});
```

Compare with `jest.mock("./api")` — that replaces the entire function, skipping fetch, URL construction, and response handling. You're testing the mock, not your code.

## Where to Test What

| Level | Test these | Don't test these |
|---|---|---|
| Unit | Reducers, formatters, validators, pure hooks | Components, DOM, side effects |
| Integration | Component rendering, user interactions, form submissions, error states | Internal state, CSS classes, render counts |
| E2E | Login, checkout, signup — critical multi-page journeys | Individual components, edge cases |

## Testing Forms

```jsx
test("validates email on submit", async () => {
  render(<SignupForm />);
  await userEvent.type(screen.getByLabelText(/email/i), "not-an-email");
  await userEvent.click(screen.getByRole("button", { name: /sign up/i }));
  expect(screen.getByText(/invalid email/i)).toBeInTheDocument();
});
```

Test each validation timing path separately: blur validation needs `userEvent.tab()`, real-time validation asserts after each `type`, submit validation fills all fields then clicks.

## Testing Error Boundaries

Create a component that throws during render. Mock `console.error` to suppress noise. Assert the fallback UI is present. Always restore the mock.

```jsx
function ThrowingComponent() { throw new Error("Test error"); }

test("error boundary catches render errors", () => {
  const spy = jest.spyOn(console, "error").mockImplementation(() => {});
  render(<ErrorBoundary><ThrowingComponent /></ErrorBoundary>);
  expect(screen.getByText(/something went wrong/i)).toBeInTheDocument();
  spy.mockRestore();
});
```

## Q&A

**1. Why test behavior instead of implementation?**

Implementation tests break on refactors that don't change behavior. Renaming a state variable or swapping a CSS class shouldn't break a test. Behavior tests survive refactors because they assert what the user sees and does.

**2. Why mock with MSW instead of jest.mock?**

MSW tests the full code path: component → hook → fetch → response handling. jest.mock replaces the entire function, skipping steps. If the API response shape changes, the mock still returns the old shape — test passes, production breaks.

**3. When should I use renderHook vs testing through a component?**

Use renderHook for pure logic hooks (reducers, transformers). Test through a component when the hook orchestrates side effects (fetching, subscriptions). The component is the real consumer boundary.

**4. How do I avoid brittle snapshot tests?**

Don't snapshot entire component trees. Snapshot tests assert structure, not behavior. If a snapshot breaks on every CSS change, it's testing the wrong thing. Use targeted assertions on specific elements and text instead.

## Mental Trigger

**Test what the user sees and does. Mock boundaries you don't own.**
